# TuneWeave 实施顺序与持续上游同步

本文件约束 TuneWeave 的实际迁移顺序。若简化说明与项目最初需求冲突，以最初需求为准：最终仍须完整覆盖参考项目范围内的全部 API、全部参数和全部行为分支；阶段排序只决定先后，不构成删减理由。

## 阶段顺序

1. 网易云 Basic
2. QQ 音乐 Basic
3. B 站 Basic
4. 网易云全量
5. QQ 音乐全量
6. B 站范围内全量：播放、列表、账户以及音视频直接相关能力；专栏、直播等无关业务不纳入
7. 酷狗、咪咕、酷我；酷我在接入前先逐项验证可用度

每个平台的 Basic 完成后才切换到下一平台 Basic。三个 Basic 阶段完成后，再依次回到各平台全量阶段。已经实现的评论能力继续保留；网易云评论读取、写入、反应目录、点赞、统计和举报这一连续小块收口后，剩余社交能力延后到网易云全量阶段。

## Basic 范围与优先级

Basic 不是按模块文件名字母排序，而是按普通音乐 App 的使用频率、播放链路依赖和用户价值排序：

1. 搜索与发现：单曲、专辑、歌手、歌单、MV/视频、歌词、播客/电台搜索，搜索建议，首页推荐、榜单和横幅。
2. 内容展示：歌曲、专辑、歌手、歌单、歌词、MV/视频、播客/电台及节目的详情和完整列表。
3. 播放与权益：可听性、全部音质、试听窗口、VIP/数字专辑权益、VIP 播放地址、MV/视频流、播客音频、播放失败后的严格跨平台匹配和回退。
4. 账户与身份：二维码、账号密码、手机号密码、验证码、登录状态、刷新、退出、国家区号、账户资料和 VIP 状态。
5. 个人音乐库：喜欢、收藏、关注歌手、创建/维护歌单、播放历史、每日推荐、私人 FM、云盘上传/导入/匹配/播放。
6. 平台基础协议：原始 API、批量 API、全部加密协议、设备身份、匿名 token、实时 checkToken、随机中国 IP、代理/真实 IP 的安全服务端配置以及平台所需签名。

网易云 Basic 的剩余实现中，云盘优先级高于播客：先把上传/直传/导入/匹配与云盘列表、详情、删除、源文件下载、直接播放串成完整可用链路，再实现播客/电台节目的目录、详情和播放链。`song_cloud_download` 属于云盘链路，不能因名称位于歌曲模块而延后。

社交动态、私信、粉丝关系、评论反应写入、一起听等不属于 Basic；它们仍在对应平台全量账本中逐项实现。播放权益相关广告、免费听、VIP、云盘、MV 和播客属于 Basic，不按“边缘功能”延后。

网易云 Basic 的可计算阶段进度见 [`docs/coverage/netease-basic.md`](coverage/netease-basic.md)。该聚合账本只用于阶段验收和排序；407 项逐模块全量状态仍以 [`docs/coverage/netease.md`](coverage/netease.md) 为准。

## Uni Playlist

Uni Playlist 是 TuneWeave 自有的跨平台歌单，不归属于任一音乐平台，公开引用使用 `uni:<opaque-id>`。它必须支持：

- 从任意已注册平台导入一个歌单，并完整遍历上游分页；导入项保留原始平台引用、顺序、重复项、内容类型和必要元数据快照。
- 手工添加任意平台的可播放内容；首批内容类型至少包含歌曲、MV/视频音频和播客/电台节目，不把所有资源强压成无类型字符串。
- 同一歌单混合网易云、QQ、B 站及后续平台内容；移除、重排和去重由调用方明确选择，不静默改写。
- 播放时先尝试条目的原始平台；指定 `playback_platform` 或原平台失败时，使用标题、歌手、专辑、时长、ISRC 和版本标签进行严格匹配，再按有序平台策略回退。
- 账户选择按目标平台分别解析，不能让导入平台的单一 `account` 别名错误覆盖其他平台登录态。
- 持久化与 provider 解耦，使用紧凑的单文件存储后端；媒体字节和平台凭据不写入歌单数据库。

计划统一端点：

| 方法 | 端点 | 用途 |
| --- | --- | --- |
| POST | `/v1/uni/playlists` | 创建空 Uni Playlist |
| POST | `/v1/uni/playlists/imports` | 从任意平台歌单创建导入快照 |
| GET | `/v1/uni/playlists/{ref}` | 读取元数据 |
| GET | `/v1/uni/playlists/{ref}/items` | 分页读取混合内容 |
| POST | `/v1/uni/playlists/{ref}/items` | 添加带类型的平台资源引用 |
| DELETE | `/v1/uni/playlists/{ref}/items/{item_id}` | 删除指定条目 |
| PATCH | `/v1/uni/playlists/{ref}/items/order` | 显式重排 |

实现时还需给现有 `/v1/playlists` 统一读取层增加 `uni` provider/存储适配，保证客户端不必维护第二套播放模型。

## 上游更新节奏

- 每完成 3 个可独立验收的上游 API 模块，fetch 当前活跃平台参考仓库并比较固定 SHA；发现新增模块或行为变更时，先更新覆盖账本再继续实现。
- 每次切换 Basic 平台、进入全量阶段、准备发布前，检查全部 8 个参考仓库。
- 同步只允许干净工作树上的 fast-forward；非快进、删除模块或许可证变化必须人工审阅差异，不得用 reset 覆盖。
- `docs/upstream-audit.md` 记录固定提交、日期、模块总数和增量影响；各覆盖账本必须同步增删条目并重新计算状态数。
- 新增接口默认进入 `pending`，不得因为刚出现、低频或平台特有而遗漏；已实现接口若上游变化则退回 `partial`，直到新行为完成测试和真实验证。

## 提交、推送与验证

每个完整小功能按核心契约、平台适配、HTTP 暴露、文档/账本拆成可审阅的小提交。每块均执行格式化、测试、严格 Clippy 和编译。远端使用逐提交 CI 闸门：只有当前 `origin/main` 对应 Actions 成功后，才推送下一个排队提交。普通 push 和手动构建不发布；只有明确 release push 或手动 release 触发才生成 Release。
